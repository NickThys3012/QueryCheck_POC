# Fully Automatic Copilot Agent Trigger — Analysis

## The problem

The issue template sets `assignees: ["copilot"]`, but this **does not automatically start the agent**. It only pre-fills the assignee field. The developer still has to:

1. Open the issue → see Copilot in assignee field
2. Click the assignee to confirm → triggers agent
3. Pick `sql-reviewer` from the custom agent dropdown → **extra manual step**

The goal: dev opens an issue → Copilot starts automatically with the `sql-reviewer` agent, **zero clicks**.

---

## Why `GITHUB_TOKEN` can't do this

GitHub requires a **user token** to trigger the Copilot coding agent via API. The default `GITHUB_TOKEN` in Actions is an installation token — it cannot trigger the agent. This is a hard platform restriction, not a config issue.

The two valid user token types are:
- **Personal Access Token (PAT)** — tied to a specific GitHub user account
- **GitHub App user-to-server token** — generated by a GitHub App acting on behalf of a user

---

## Option A: Fine-grained PAT

The simplest approach. One person creates a PAT, stores it as a repo secret, and the workflow uses it.

### Setup (one-time)

**1. Create the PAT**

Go to: GitHub → Settings → Developer settings → Fine-grained personal access tokens → Generate new token

Required permissions on the `QueryCheck_POC` repository:
- Read and write: **Actions, Contents, Issues, Pull Requests**
- Read only: **Metadata**

**2. Store it as a repo secret**

Repository → Settings → Secrets and variables → Actions → New repository secret:
- Name: `COPILOT_PAT`
- Value: the token you just created

**3. Remove the assignee from the issue template**

Since the workflow assigns Copilot programmatically, remove `assignees: ["copilot"]` from `.github/ISSUE_TEMPLATE/sql-review.yml` to avoid conflicts.

### How it works

```
Dev opens issue using the sql-review template
    → GitHub Actions workflow triggers (issues: opened, label: sql-review)
    → Workflow calls REST API with COPILOT_PAT:
         - assigns copilot-swe-agent[bot] to the issue
         - sets custom_agent: "sql-reviewer"
    → Copilot coding agent starts automatically
    → Agent reads sql-reviewer.agent.md + skill rules
    → Posts reformatted SQL + analysis as issue comment
    → Reviewer reads comment, approves or requests changes
```

### Workflow file (REST API — simpler than GraphQL)

```yaml
name: SQL Auto-Assign to Copilot

on:
  issues:
    types: [opened]

jobs:
  assign:
    if: contains(github.event.issue.labels.*.name, 'sql-review')
    runs-on: ubuntu-latest

    steps:
      - name: Assign Copilot with sql-reviewer agent
        env:
          GH_TOKEN: ${{ secrets.COPILOT_PAT }}
        run: |
          gh api \
            --method POST \
            -H "Accept: application/vnd.github+json" \
            -H "X-GitHub-Api-Version: 2022-11-28" \
            /repos/${{ github.repository }}/issues/${{ github.event.issue.number }}/assignees \
            --input - <<< '{
              "assignees": ["copilot-swe-agent[bot]"],
              "agent_assignment": {
                "target_repo": "${{ github.repository }}",
                "custom_agent": "sql-reviewer"
              }
            }'
```

> **Note**: The REST API (`/issues/{number}/assignees`) is simpler than GraphQL for this use case — no need to look up bot IDs or repo GraphQL IDs. Both achieve the same result.

### Pros & Cons

| | |
|--|--|
| ✅ Simplest to set up | ❌ Token tied to one person's GitHub account |
| ✅ No infrastructure needed | ❌ Must be rotated manually when it expires |
| ✅ Works immediately | ❌ If that person leaves, token stops working |
| ✅ Uses actual Copilot agent + sql-reviewer | |

**Best for**: small teams, personal repos, or PoC — get it working fast.

## Option B: GitHub App (no personal dependency)

A GitHub App generates tokens automatically and is not tied to any individual user account. Better for team or org use.

### Setup (one-time, more involved)

**1. Register a GitHub App**

GitHub → Settings → Developer settings → GitHub Apps → New GitHub App

Fill in:
- Name: e.g. `QueryCheck SQL Reviewer`
- Homepage URL: your repo URL
- Uncheck "Active" under Webhooks (not needed)

Permissions required (Repository):
- Actions: Read & write
- Contents: Read & write
- Issues: Read & write
- Pull Requests: Read & write
- Metadata: Read only

**2. Generate a private key**

On the app settings page → "Private keys" → Generate a private key. Download the `.pem` file.

**3. Install the app on the repository**

GitHub App settings → Install App → select `QueryCheck_POC` repo.

Note the **Installation ID** from the URL after installing (e.g. `https://github.com/settings/installations/12345678`).

**4. Store secrets in the repo**

Repository → Settings → Secrets and variables → Actions:

| Secret name | Value |
|--|--|
| `APP_ID` | The App ID shown on the app settings page |
| `APP_PRIVATE_KEY` | Full contents of the downloaded `.pem` file |
| `APP_INSTALLATION_ID` | The Installation ID from step 3 |

**5. Remove the assignee from the issue template** (same as Option A)

### Workflow file

```yaml
name: SQL Auto-Assign to Copilot

on:
  issues:
    types: [opened]

jobs:
  assign:
    if: contains(github.event.issue.labels.*.name, 'sql-review')
    runs-on: ubuntu-latest

    steps:
      - name: Generate GitHub App user token
        id: app-token
        uses: actions/create-github-app-token@v1
        with:
          app-id: ${{ secrets.APP_ID }}
          private-key: ${{ secrets.APP_PRIVATE_KEY }}
          # user-to-server token is required to trigger the Copilot agent
          # installation tokens (server-to-server) do NOT work for this

      - name: Assign Copilot with sql-reviewer agent
        env:
          GH_TOKEN: ${{ steps.app-token.outputs.token }}
        run: |
          gh api \
            --method POST \
            -H "Accept: application/vnd.github+json" \
            -H "X-GitHub-Api-Version: 2022-11-28" \
            /repos/${{ github.repository }}/issues/${{ github.event.issue.number }}/assignees \
            --input - <<< '{
              "assignees": ["copilot-swe-agent[bot]"],
              "agent_assignment": {
                "target_repo": "${{ github.repository }}",
                "custom_agent": "sql-reviewer"
              }
            }'
```

> **Important**: `actions/create-github-app-token@v1` generates an **installation token** (server-to-server), which GitHub may or may not accept for triggering the Copilot agent. GitHub's docs state a **user-to-server** token is required. If the installation token is rejected, a user OAuth flow via the app would be needed — significantly more complex. This is the main uncertainty with Option B.

### Pros & Cons

| | |
|--|--|
| ✅ Not tied to any individual | ❌ More complex setup |
| ✅ Token auto-rotates | ❌ User-to-server token requirement may still need a human to authorize |
| ✅ Better for org/team use | ❌ Installation token may not work (unconfirmed) |
| ✅ Follows security best practices | |

**Best for**: production, team-owned repos, or when the org policy prohibits PATs.

## One-time verification step (before implementing either option)

Before building the workflow, verify the Copilot agent is visible in the repo. Run this with a user token that has repo access:

```bash
gh api graphql -f query='
query {
  repository(owner: "NickThys3012", name: "QueryCheck_POC") {
    suggestedActors(capabilities: [CAN_BE_ASSIGNED], first: 10) {
      nodes {
        login
        __typename
        ... on Bot { id }
      }
    }
  }
}' -H 'GraphQL-Features: issues_copilot_assignment_api_support'
```

If the output includes a node with `login: "copilot-swe-agent"` → the agent is available and enabled.  
If not → Copilot cloud agent needs to be enabled in the repo settings first (Settings → Copilot → Coding agent).

---

## Recommendation

**Start with Option A (PAT)**:

1. It's the fastest to validate — if the API works, you know Option B will too
2. The Copilot agent assignment API is still in public preview — worth proving it works before investing in GitHub App infrastructure
3. Once confirmed working and the team grows, migrate to Option B

Both options keep the current GitHub Actions + Models API flow running in parallel — it still handles re-reviews via `/re-review` comments, which the Copilot agent approach doesn't support natively.
